### 1. Double `expf` computation in KDA path — **HIGH IMPACT, XS EFFORT**

**File:** `gated_delta_net_chunk.cu:126-143`

In the KDA branch, `expf(g_shard[r])` is computed twice per token per row — once for the `kv_shard` accumulation and again for the state update. This is pure redundant compute.

```cuda
// First computation (line ~131):
kv_shard += expf(g_shard[r]) * s_shard[r] * k_shard[r];

// Second computation (line ~139):
s_shard[r] = expf(g_shard[r]) * s_shard[r] + k_shard[r] * delta_col;
```

**Fix:** Cache `expf(g_shard[r])` in a register array, identical to how the non-KDA path already caches `g_val`:

```cuda
// KDA: per-feature gate
const float * g_t = g + gb * S_v;
float eg_shard[rows_per_lane];  // cached exp(g)
#pragma unroll
for (int r = 0; r < rows_per_lane; r++) {
    const int i = r * warp_size + lane;
    eg_shard[r] = (i < S_v) ? expf(g_t[i]) : 0.0f;

float kv_shard = 0.0f;
#pragma unroll
for (int r = 0; r < rows_per_lane; r++) {
    kv_shard += eg_shard[r] * s_shard[r] * k_shard[r];

float kv_col = warp_reduce_sum<warp_size>(kv_shard);
float delta_col = (v_col - kv_col) * beta_val;

float attn_partial = 0.0f;
#pragma unroll
for (int r = 0; r < rows_per_lane; r++) {
    s_shard[r] = eg_shard[r] * s_shard[r] + k_shard[r] * delta_col;
    attn_partial += s_shard[r] * q_shard[r];

```

For S_v=128, rows_per_lane=4 (warp_size=32) or 2 (warp_size=64). This saves 4 or 2 `expf` calls per thread per token. On a 4K-token prefill with H=32 heads, that's hundreds of thousands of avoided transcendentals.

### 2. Redundant global memory loads for `beta` and `g` — **HIGH IMPACT, S EFFORT**

**File:** `gated_delta_net_chunk.cu:98-100` (non-KDA), `gated_delta_net_chunk.cu:117-124` (KDA)

Every thread in the block independently loads `beta[gb]` and (in non-KDA) `g[gb]`. With 128 threads per block, this is 128 identical loads to the same address. While L1 cache will absorb most of these, it still generates redundant memory transactions and wastes instruction slots.

**Fix:** Load once in lane 0 and broadcast via `__shfl_sync`:

```cuda
// Non-KDA path:
float beta_val, g_val;
if (lane == 0) {
    beta_val = beta[gb];
    g_val = expf(g[gb]);

beta_val = __shfl_sync(0xffffffff, beta_val, 0);
g_val = __shfl_sync(0xffffffff, g_val, 0);
```

For KDA, `beta[gb]` should be broadcast the same way. The per-feature `g_t[i]` loads are already distributed across lanes and should remain as-is.

### 3. No shared memory caching for chunk-local q/k — **HIGH IMPACT, M EFFORT**

**File:** `gated_delta_net_chunk.cu:88-103`

Each token's `q` and `k` vectors (S_v floats each) are loaded from global memory every iteration of the inner token loop. For CS=64 tokens with S_v=128, that's 64 × 128 × 2 = 16,384 floats (64 KB) loaded per chunk. If q/k are accessed by multiple warps (they are — all warps in the block read the same q_t and k_t), shared memory can serve them after a single global load.

**Current pattern:**
```cuda
for (int p = 0; p < n_local; p++) {
    // All 4 warps load the same k_t[i] and q_t[i]
    k_shard[r] = k_t[i];
    q_shard[r] = q_t[i];

```

**Fix:** Allocate shared memory for one token's q and k (2 × S_v floats = 512 bytes for S_v=128). Load cooperatively at the start of each token iteration, then read from shared memory:

```cuda
__shared__ float s_q[128];  // sized for max S_v
__shared__ float s_k[128];

for (int p = 0; p < n_local; p++) {
    const int t = t_base + p;
    const float * k_t = k + iq3 * sq3 + t * sq2 + iq1 * sq1;
    const float * q_t = q + iq3 * sq3 + t * sq2 + iq1 * sq1;

    // Cooperative load: each thread loads one element
    const int tid = threadIdx.y * blockDim.x + threadIdx.x;
    if (tid < S_v) {
        s_k[tid] = k_t[tid];
        s_q[tid] = q_t[tid];
   syncthreads();

    // Read from shared memory
    #pragma unroll
    for (int r = 0; r < rows_per_lane; r++) {
        const int i = r * warp_size + lane;
        k_shard[r] = s_k[i];
        q_shard[r] = s_q[i];

    // ... rest of computation ...
    __syncthreads();  // before next iteration's load

```

This reduces global memory reads for q/k by a factor of `num_warps` (4×). For S_v=128, CS=64, this saves ~48 KB of global reads per chunk.

**Caveat:** For S_v=128 with CS=64, double-buffering the shared memory (load next token's q/k while computing current) would hide the load latency. This requires 2 × 2 × S_v = 1024 bytes of shared memory — trivial.

### 4. Conservative `__launch_bounds__` limiting occupancy — **MEDIUM IMPACT, XS EFFORT**

**File:** `gated_delta_net_chunk.cu:19`

```cuda
__launch_bounds__((ggml_cuda_get_physical_warp_size() < S_v ? ggml_cuda_get_physical_warp_size() : S_v) * 4, 2)
```

The second parameter `2` tells the compiler to optimize for at least 2 blocks per SM. With 128 threads per block (warp_size=32, num_warps=4), that's 256 threads = 8 warps per SM. On a modern GPU with 64+ warps per SM, this is ~12% occupancy.

With ~20 VGPRs per thread (as stated), the hardware could support far more blocks:
- 65,536 VGPRs per SM ÷ (20 VGPRs × 128 threads) = 25 blocks per SM (register-limited)
- Shared memory: none used, so not a constraint

**Fix:** Increase the min-blocks hint to 4 or 8:

```cuda
__launch_bounds__((ggml_cuda_get_physical_warp_size() < S_v ? ggml_cuda_get_physical_warp_size() : S_v) * 4, 4)
```

This allows the compiler to allocate slightly more registers if needed while still enabling 4 blocks/SM (16 warps, ~25% occupancy). Profile with `nvprof`/`rocprof` to find the sweet spot — occupancy beyond 25% often yields diminishing returns for compute-bound kernels, but 12% is almost certainly too low.

### 5. Wavefront underutilization on AMD gfx906 — **MEDIUM IMPACT, M EFFORT**

**File:** `gated_delta_net_chunk.cu:19, 62-63`

```cuda
constexpr int warp_size = ggml_cuda_get_physical_warp_size() < S_v ? ggml_cuda_get_physical_warp_size() : S_v;
```

On gfx906 (Vega 20 / MI50), the physical wavefront size is 64. For S_v=16 or S_v=32, `warp_size` is clamped to S_v (16 or 32). This means only 16 or 32 threads are active in a 64-thread wavefront, wasting 50–75% of the wavefront.

**Impact:** For S_v=16, each wavefront has 48 idle threads. The GPU still schedules full wavefronts, so you're paying for 64 threads but using 16.

**Fix:** For S_v < physical_warp_size, pack multiple columns into a single wavefront. Instead of `threadIdx.y` indexing separate warps, use sub-wavefront groups:

```cuda
// For S_v=16 on wave64: 4 columns per wavefront (64/16=4)
constexpr int cols_per_wave = ggml_cuda_get_physical_warp_size() / S_v;
const int sub_col = threadIdx.x / S_v;  // which column within the wavefront
const int lane = threadIdx.x % S_v;     // lane within the sub-group
```

This requires restructuring the reduction to use sub-wavefront shuffles (`__shfl_down` with appropriate masks) but would fully utilize the wavefront. The `warp_reduce_sum` would need a width parameter:

```cuda
float kv_col = warp_reduce_sum<S_v>(kv_shard);  // reduce within sub-group of S_v lanes
```

**Effort:** Medium — requires careful testing across S_v values and both NVIDIA/AMD paths.

### 6. Output write pattern: only lane 0 writes, poor coalescing — **MEDIUM IMPACT, S EFFORT**

**File:** `gated_delta_net_chunk.cu:107-109, 147-149`

```cuda
if (lane == 0) {
    attn_data[col] = attn_col * scale;

```

Only lane 0 of each warp writes the result. With 4 warps per block, that's 4 writes per block per token, issued by different warps at different times. The writes are to `col, col+1, col+2, col+3` (consecutive addresses), but they're not issued together, so they don't coalesce into a single transaction.

**Fix:** Use shared memory to collect results, then have a single warp write them coalesced:

```cuda
__shared__ float s_attn[4];  // num_warps

if (lane == 0) {
    s_attn[threadIdx.y] = attn_col * scale;

__syncthreads();

// Lane 0 of warp 0 writes all 4 values
if (threadIdx.x == 0 && threadIdx.y == 0) {
    for (int w = 0; w < num_warps; w++) {
        attn_data[col - threadIdx.y + w] = s_attn[w];
   ```

Or better, use a single `float4` store if the 4 values are consecutive and aligned.

### 7. Inner token loop not unrolled — **LOW-MEDIUM IMPACT, XS EFFORT**

**File:** `gated_delta_net_chunk.cu:85`

```cuda
for (int p = 0; p < n_local; p++) {
```

`n_local` is runtime-variable (depends on whether this is the last chunk). The compiler cannot unroll this loop. For the common case where `n_local == CS`, the loop body executes 64 (or 16) times with no unrolling.

**Fix:** Split into a fast path (full chunk) with `#pragma unroll` and a slow path (partial chunk):

```cuda
if (n_local == CS) {
    #pragma unroll 4  // or higher, tune with profiler
    for (int p = 0; p < CS; p++) {
        // ... loop body ...
    {
    for (int p = 0; p < n_local; p++) {
        // ... loop body ...
   ```

Unrolling by 4 reduces loop overhead and enables instruction-level parallelism between iterations (e.g., overlapping loads from iteration p+1 with compute from iteration p).

### 8. No vectorized (float4) loads for q/k — **LOW-MEDIUM IMPACT, S EFFORT**

**File:** `gated_delta_net_chunk.cu:93-96`

```cuda
for (int r = 0; r < rows_per_lane; r++) {
    const int i = r * warp_size + lane;
    k_shard[r] = k_t[i];
    q_shard[r] = q_t[i];

```

Each thread loads `rows_per_lane` individual floats. If the base address is 16-byte aligned and the stride permits, these could be loaded as `float2` or `float4` for higher bandwidth utilization.

For S_v=128, warp_size=32, rows_per_lane=4: each thread loads 4 floats at indices `[lane, lane+32, lane+64, lane+96]`. These are not contiguous per-thread, so `float4` doesn't apply directly. However, if you restructure to load 4 consecutive elements per thread (indices `[lane*4, lane*4+1, lane*4+2, lane*4+3]`) using `float4`, then redistribute via `__shfl_sync`, you can achieve vectorized loads.

**Caveat:** This restructuring changes the data layout in registers and requires corresponding changes to the reduction and state update logic. The benefit is modest (~10-15% bandwidth improvement) and may not be worth the complexity unless profiling shows memory bandwidth as the bottleneck.

### 9. State initialization doesn't exploit coalescing — **LOW IMPACT, XS EFFORT**

**File:** `gated_delta_net_chunk.cu:68-73`

```cuda
const float * state_in = curr_state + state_in_offset + col * S_v;
// ...
for (int r = 0; r < rows_per_lane; r++) {
    const int i = r * warp_size + lane;
    s_shard[r] = (i < S_v) ? state_in[i] : 0.0f;

```

`state_in` is indexed as `state_in[col * S_v + i]`. Within a warp, all lanes access the same `col` (since col depends on `threadIdx.y`, not `threadIdx.x`), so `state_in[i]` for different lanes accesses consecutive addresses. This is coalesced — good.

However, across warps (different `threadIdx.y`), the accesses are to different `col` values, which are consecutive columns. Since the state is stored column-major (`col * S_v + row`), consecutive columns are strided by `S_v` floats. This is not coalesced across warps, but since each warp issues its own memory transactions, this is acceptable.

**No fix needed** — this is already correct.

### 10. Algorithmic: Chunk-internal parallelism via associative scan — **HIGH IMPACT, XL EFFORT**

The current kernel processes tokens within a chunk sequentially (lines 85–155). The recurrence:

```
S_t = g_t * S_{t-1} + k_t * delta_t^T
```

is a linear recurrence that can be parallelized using a work-efficient parallel scan (Blelloch scan) if the state updates are expressed as associative operations. This would allow processing CS tokens in O(log CS) parallel steps instead of O(CS) sequential steps.

**Effort:** XL — requires reformulating the recurrence as a scan operation, implementing a shared-memory scan kernel, and handling the non-commutative nature of the state update. This is a research-level change, not a local optimization.

**Reference:** The "Parallelizing Linear RNNs" literature (e.g., parallel scan for SSMs like Mamba) demonstrates this approach. If the chunk size is large enough (CS=64), the O(log 64)=6 parallel steps vs. 64 sequential steps could yield significant speedup on large GPUs.
