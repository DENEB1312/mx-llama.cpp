## 1. Grounded observations (before the plan)

These matter because gfx906 is wavefront-64, not 32, and the kernel's existing warp-size adaptation is load-bearing for every optimization below:

- `gated_delta_net.cu:48` defines `warp_size = min(physical_warp, S_v)`. On gfx906 (`physical_warp=64`) this gives `{S_v=16→16, 32→32, 64→64, 128→64}` and `rows_per_lane = {1, 1, 1, 2}` respectively (`gated_delta_net.cu:49-50`).
- Each warp owns **one column** of S and reduces across rows via `warp_reduce_sum<warp_size>` (`gated_delta_net.cu:99`, `:111`, `:127`, `:138`). For S_v=128 on wave64, each lane holds 2 floats of that column — every reduction is over 64 lanes.
- The state matrix S is stored **transposed** (`M[col][i] = S[i][col]`, `gated_delta_net.cu:54`), so the per-lane `s_shard[r]` accesses are stride-1 in memory on load/store. This is already the right layout — do not re-layout.
- Launch bounds are `(warp_size*4, 2)` (`gated_delta_net.cu:4-5`) → 4 warps/block, ≥2 blocks/CU. With 256 threads/block at S_v=128 on wave64, that's 512 threads/CU minimum. MI50 has 256 SIMD lanes per CU (4 SIMDs × 64); occupancy is fine but register file (256 VGPRs/lane) is the ceiling to watch.
- The `attn_data[col] = …` write is lane-0-only (`gated_delta_net.cu:113, 139`). On wave64, 63/64 lanes idle on that store — acceptable because it's one scalar per token, but worth noting if you ever vectorize the token loop.

## 2. Prioritized optimization plan

Sorted by **Impact × (1/Effort)**. Effort: XS/S/M/L. Impact is relative throughput gain on the whole kernel for realistic workloads (n_tokens ∈ {1, 64, 2048}, S_v ∈ {64, 128}, KDA on/off).

### #1 — Cache `expf(g_t[i])` in the KDA branch (XS effort, HIGH impact)

**File / lines:** `gated_delta_net.cu:119-145`.

**What is actually wrong:** In the KDA branch, `expf(g_t[i])` is computed **twice per row per token** — once for the `kv_shard` accumulation (`gated_delta_net.cu:123`) and again for the state update (`gated_delta_net.cu:134`). For S_v=128 on wave64 (2 rows/lane), that is 4 redundant `expf` calls per token per column. `expf` on gfx906 is ~20–30 cycles with a dependent FMA chain; this is the single largest arithmetic cost in the KDA path.

**Change (diff-shaped):**
```cpp
// inside the `else /* KDA */` branch, before the kv_shard loop:
float g_exp[rows_per_lane];
#pragma unroll
for (int r = 0; r < rows_per_lane; r++) {
    const int i = r * warp_size + lane;
    g_exp[r] = expf(g_t[i]);

// then replace expf(g_t[i]) with g_exp[r] on lines 123 and 134:
//   kv_shard += g_exp[r] * s_shard[r] * k_reg[r];
//   s_shard[r] = g_exp[r] * s_shard[r] + k_reg[r] * delta_col;
```

**Register cost:** +`rows_per_lane` VGPRs (≤2 on gfx906 at S_v=128). Trivial vs. the 256-VGPR budget.

**Expected speedup:** KDA-token inner loop ≈ 1.5–1.8×. Non-KDA path unchanged. End-to-end for a KDA-heavy model (e.g., Qwen3-next / Kimi-Linear): ~1.3–1.5× on the fused GDN op.

**Risk:** None. Purely algebraic identity. Verify with a bitwise-tolerant numerical test against the CPU reference (`ops.cpp:10509-10709`) — FP ordering changes slightly, so allow atol=1e-4 on logits.

**Incremental?** Yes. One commit, no other change required.

### #2 — Eliminate dead state writes in the `keep_rs_t=true` path (S effort, HIGH impact for prefill)

**File / lines:** `gated_delta_net.cu:148-164` (inside the per-token loop) and `:28-29` (grid).

**What is actually wrong:** When `keep_rs_t==true`, the kernel writes the full S_v² state for every token whose `target_slot = t - (n_tokens - K)` lands in `[0, K)`. That is, it writes for the **last K tokens** of the sequence — but the `if` is evaluated on every iteration of `t`. For K=4 and n_tokens=2048, that means 2044 iterations evaluate a false branch (cheap) but the last 4 iterations each write S_v²×H×n_seqs floats (very expensive). The cost is not the branch — it is the writes themselves: for S_v=128, H=32, K=4, n_seqs=1, that is 4 × 32 × 16 384 × 4 B = **8 MiB** of HBM writes per forward pass, serialized per column.

**Two concrete fixes, pick based on workload:**

(a) **Precompute the start token** and skip the `if` entirely:
```cpp
const int shift = (int)n_tokens - K;
const int t_write_start = max(0, shift);
// … main loop processes t in [0, n_tokens) …
// After the loop (or inside it, guarded by `t >= t_write_start`), write state
// into dst + (t - shift) * state_size_per_token + state_out_offset.
```
This removes the per-iteration branch and makes the write region explicit. Small win on its own.

(b) **Register-resident state across tokens; write only at snapshot boundaries.** The current kernel already keeps `s_shard` in registers across tokens — the writes come from those same registers. The cost is purely the HBM stores. For small K (K≤4 is typical, see `delta-net-base.cpp:262` where `K = cparams.n_rs_seq + 1`), consider **not writing intermediate snapshots** and instead recompute them in a second small kernel, or have the host side consume only the final state. This requires a contract change with `build_recurrent_attn` (`delta-net-base.cpp:548-605`) and is not a pure kernel change — see Risk below.

**Expected speedup:** (a) ~1.05–1.1× on prefill, mostly from removing a taken branch in the hot loop tail. (b) ~1.3–1.6× on prefill with K>1 (eliminates ¾ of state traffic at K=4), but requires host-side changes.

**Risk:** (a) none. (b) medium — the slot-mapping is consumed by `build_recurrent_attn` which copies each snapshot to `ssm_states_all` (`delta-net-base.cpp:591-603`). If you drop intermediate snapshots, the ring-buffer semantics in `llama-memory-recurrent` must be updated, and any code path that reads slot `k_i < K-1` will silently get stale data. Do not do (b) without auditing all callers of `build_recurrent_attn` with `keep=true`.

**Incremental?** (a) yes. (b) no — it couples to `delta-net-base.cpp` and the memory subsystem.

### #3 — Chunked pre-fill kernel (L effort, VERY HIGH impact)

**File / lines:** `gated_delta_net.cu:175` (the TODO) and the dispatch in `delta-net-base.cpp:399-405` (`build_delta_net_fused` always calls the per-token kernel regardless of `n_tokens`).

**What is actually wrong:** The current kernel is inherently sequential in `t`: each token depends on the state left by the previous token. For `n_tokens=2048`, that is 2048 iterations per column, with two warp-reduces and a state update per iteration. The chunking math is **already implemented on the CPU graph side** in `build_delta_net_chunking` (`delta-net-base.cpp:28-238`) and expresses the recurrence as:

- Intra-chunk: triangular solve against a `(CS × CS)` decay-masked kernel (`delta-net-base.cpp:146-165`).
- Inter-chunk: a single state update per chunk, using `g_last`, `k_cd`, and `kg_t` (`delta-net-base.cpp:195-225`).

With `CS=64` (non-KDA) or `CS=16` (KDA, see `delta-net-base.cpp:77`), a 2048-token prefill becomes 32 or 128 chunk-level updates instead of 2048 token-level updates. The intra-chunk work is a `ggml_mul_mat` on a small matrix — perfectly suited for a MFMA/X matrix core dispatch on gfx906 (`v_mfma_f32_32x32x1f32` etc.).

**Concrete plan (two kernels, shippable independently):**

- **Kernel A — intra-chunk:** launch one block per `(head, seq, chunk)`. Inputs: `q,k,v,g,beta` slices of size `CS`. Computes the closed-form chunk output `o_intra` and the chunk state delta `ΔS_chunk` using the same math as `build_delta_net_chunking` lines 77-193. Output: `o_intra` (additive contribution to final output) and `ΔS_chunk` (to be applied to running state).
- **Kernel B — inter-chunk state sweep:** sequential over chunks (or a small log-depth scan), applies `S ← g_last·S + ΔS_chunk` and produces the per-token `o_inter = S · q`. For `keep_rs_t`, also snapshots state at the last K chunk boundaries.

**Expected speedup:** For `n_tokens ≥ 128` on S_v=128 KDA, **3–6×** over the current per-token kernel, driven by (i) replacing 2 warp-reduces/token with one MFMA/chunk and (ii) amortizing state reads over CS tokens. The non-KDA path benefits less (already cheaper per token) — expect ~1.5–2.5×.

**Risk:** High if done naïvely.
- Numerical: the chunked form uses `exp(cumsum(g))` and a triangular solve. The solve is the stability risk — diagonal dominance depends on `beta ∈ [0,1]` and decay < 1. The CPU reference already does this (`delta-net-base.cpp:157-165`, `ggml_solve_tri`), so you have a ground-truth comparator.
- Correctness on gfx906 specifically: MFMA instructions on Vega 20 are **not available** — MFMA starts with CDNA (gfx908). On gfx906 you must use `v_dot4` / `v_fma` chains or `rocblas` SGEMM. Plan the chunked kernel around a rocBLAS call for the intra-chunk `CS×CS` matmul, not MFMA intrinsics.
- `keep_rs_t` interaction: snapshot slot mapping (currently `target_slot = t - shift`) must be re-derived at chunk granularity. The last K tokens may span a chunk boundary; handle partial-chunk snapshots explicitly.

**Incremental?** Yes, as an **additive** kernel dispatched from `launch_gated_delta_net` when `n_tokens ≥ CS`. The existing per-token kernel remains the fallback for `n_tokens < CS` and for the autoregressive (n_tokens=1) case. This is exactly the dispatch pattern already in `build_delta_net` (`delta-net-base.cpp:393-406`).

### #4 — BF16 state, FP32 accumulation (M effort, MEDIUM-HIGH impact)

**File / lines:** State load/store at `gated_delta_net.cu:55-60` and `:148-164`; accumulator type throughout the kernel body.

**What is actually wrong:** State is `S_v × S_v × H × n_seqs × K` floats — the single largest tensor. For a realistic config (S_v=128, H=32, n_seqs=4, K=4): **32 MiB** per layer, loaded once at start and (in `keep_rs_t`) written K times. HBM bandwidth on MI50 is 1024 GB/s; 32 MiB is ~30 µs of pure bandwidth. The reads and writes are not hidden by compute because the per-token inner loop has low arithmetic intensity (~4 FLOPs/byte in the non-KDA path, ~8 in KDA after fix #1).

**Change:**
- Store state as `ggml_bf16_t` in the `dst` output tensor and `src_state` input.
- Keep `s_shard[]`, `kv_shard`, `delta_col`, `attn_partial` as **FP32** in registers.
- Convert on load (`bf16→f32`) and on store (`f32→bf16`).

**Expected speedup:** 1.5–1.8× on memory-bound configs (large `n_seqs`, large K, or S_v=128 with short `n_tokens` where state load dominates). Negligible for autoregressive (n_tokens=1) with a single sequence because the state fits in L2 (32 MiB × layers does not, but per-call working set is small).

**Risk:** Medium.
- BF16 has the FP32 exponent range (no overflow on state growth), but only 8 mantissa bits. The recurrence `S ← g·S + k·δ` can accumulate rounding error over thousands of tokens. Validate against the FP32 kernel on a 2048-token sequence with a realistic model; expect ≤1e-3 absolute logit drift if the model was trained with BF16 state. If the reference model uses FP32 state internally (likely for the published DeltaNet checkpoints), this will **degrade quality** — measure perplexity before shipping.
- GGML plumbing: `src_state->type` and `dst->type` must be `GGML_TYPE_BF16` end-to-end. The CPU reference (`ops.cpp:10509-10709`) is FP32-only, so you must either add a BF16 path there or keep a separate FP32 test harness. The assertions in `ggml_cuda_op_gated_delta_net` (`gated_delta_net.cu:225-233`) do not currently restrict type, but the dispatch assumes FP32 pointers — change `const float *` to `const void *` and type-switch.

**Incremental?** Yes, as a new template parameter `<typename state_t>` added alongside the existing `<S_v, KDA, keep_rs_t>`. Keep FP32 as the default; add BF16 as an opt-in path selected at dispatch time based on `src_state->type`.

### #5 — Fuse the two `warp_reduce_sum` calls per token (S effort, LOW-MEDIUM impact)

**File / lines:** `gated_delta_net.cu:99, 111` (non-KDA) and `:127, 138` (KDA).

**What is actually wrong:** Each token does two warp reductions: one for `kv_col` (S·k) and one for `attn_col` (S·q). On wave64, each reduction is 6 `__shfl_xor` steps. The two reductions are **independent** and can be combined into one dual-value reduction if you interleave them in a single shuffle sequence.

**Change (sketch):** Pack `{kv_shard, attn_partial}` into two registers and run a single 6-step shuffle that reduces both in parallel. The second reduction is computed **after** the state update, so you cannot fully overlap — but you can precompute the `q_reg·s_shard` term before the state update and apply the `k_reg·delta_col·q_reg` correction afterwards as a scalar `delta_col * Σ k_reg[r]*q_reg[r]`, which itself is a single warp-reduce that can be hoisted above the state update.

Concretely: `attn_new = Σ (g·s + k·δ)·q = g·Σ s·q + δ·Σ k·q`. The first term uses the pre-update `Σ s·q` (which is last token's `attn_col`, so cache it across iterations); the second is `δ · dot(k, q)` — one reduction, hoisted above the state update, parallelizable with `kv_col = dot(s, k)`.

**Expected speedup:** ~1.1–1.15× (one fewer reduction per token in steady state).

**Risk:** Low, but the algebraic rewrite is easy to get wrong. Verify against the FP32 reference with atol=1e-5.

**Incremental?** Yes. Purely arithmetic.

### #6 — Avoid sub-wavefront shuffle on gfx906 when `S_v < 64` (S effort, LOW-MEDIUM impact, MI50-specific)

**File / lines:** `gated_delta_net.cu:48-49` and the `warp_reduce_sum<warp_size>` calls.

**What is actually wrong:** For S_v=16 or S_v=32 on gfx906 (wave64), the kernel caps `warp_size` at S_v and issues `warp_reduce_sum<16>` or `<32>`. On AMD hardware this typically compiles to a `ds_permute` within a sub-wavefront, which is **correct** but wastes 48 or 32 lanes of the wavefront (only 16 or 32 lanes are doing useful work). MI50's scheduler still dispatches the full wavefront — you pay for 64 lanes and use 16.

**Change:** For `S_v < 64`, pack **multiple columns per wavefront** using the lane index as `(col_within_wave, row_within_col)`. For S_v=32, each wavefront handles 2 columns; for S_v=16, 4 columns. This matches what `blockDim.y = num_warps` already suggests but isn't exploiting.

**Expected speedup:** ~1.5–2× on S_v=16/32 workloads on MI50. No effect on S_v=64/128 (already wave64-native).

**Risk:** Medium — this changes the lane-to-data mapping, so every load/store in the kernel body must be re-derived. Easy to introduce coalescing bugs. Only worth doing if your target model actually uses S_v ∈ {16, 32}; the models in `models.h` that use delta-net (`llama_model_qwen3next`, `llama_model_qwen35`, `llama_model_qwen35moe`, `llama_model_kimi_linear`) use S_v=128 in the published configs, so this optimization is **low value in practice** unless you are supporting smaller research configs.

**Incremental?** Yes, but it's a second template specialization of the inner loop, not a small diff.

## 3. What NOT to do (manufactured findings I explicitly reject)

- **"Use shared memory for q/k broadcast."** The current per-lane register load of `q_t[i]`, `k_t[i]` (`gated_delta_net.cu:73-80`) is already coalesced across lanes (they load different `i` from a contiguous row) and the values are reused only within one token iteration. Shared memory would add a load-store hop for no reuse win. Skip.
- **"Unroll the token loop."** `n_tokens` is dynamic (line 15); `#pragma unroll` would be ignored or blow up code size. Skip.
- **"Replace `expf` with `__expf`."** Fast-math `__expf` loses ~2 ULP accuracy. In KDA, the state is multiplied by `exp(g)` every token — errors compound exponentially over `n_tokens`. Do **not** do this without a measured quality check. The CPU reference uses `expf` (`ops.cpp` via `ggml_vec_*`); switching to `__expf` on GPU will produce divergent outputs after ~200 tokens.
- **"Increase `num_warps` from 4."** `blockDim.y = 4` (`gated_delta_net.cu:181`) is already tuned for 256 threads on wave64. Going to 8 warps (512 threads) on MI50's 256-lane CUs drops occupancy to 1 block/CU and increases register spill risk. Skip unless profiling shows otherwise.

## 4. Risks and correctness concerns across the whole plan

1. **gfx906 ≠ gfx908.** MI50 has no MFMA and no native wave32. Any optimization that assumes CDNA (matrix cores, wave32 mode) will silently compile to scalar code or produce wrong results. Every kernel change must be profiled on the actual hardware, not just on NVIDIA via HIP→CUDA translation.
2. **`ggml_cuda_get_physical_warp_size()` returns 64 on gfx906.** Every optimization that touches `warp_size`, `rows_per_lane`, or shuffle width must be re-validated for the S_v ∈ {16, 32} cases where `warp_size` is capped below the physical wavefront.
3. **`keep_rs_t` slot mapping is consumed outside the kernel.** Any change to when/how snapshots are written must be reconciled with `build_recurrent_attn` at `delta-net-base.cpp:582-603`. The cache-slot indexing `(K - 1 - k_i)` there assumes every slot is populated for `n_seq_tokens > 1` — partial writes from the kernel will leave garbage in unfilled slots.
4. **The `GGML_ABORT("fatal error")` at `gated_delta_net.cu:211`** fires for any `S_v` not in {16, 32, 64, 128}. If a future model uses S_v=96 or 256, the kernel will abort at runtime. Not a perf issue, but worth hardening while you're in the file.
5. **FP32 numerics in the chunked CPU reference** (`delta-net-base.cpp:28-238`) use `ggml_solve_tri` which is a forward-substitution. If your chunked GPU kernel uses a different solve order, expect bit-level divergence even in FP32. Use the CPU path as the ground-truth comparator with a tolerance, not with `==`.
