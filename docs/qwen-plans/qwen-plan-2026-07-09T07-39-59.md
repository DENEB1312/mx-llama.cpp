## Finding 1 — State matrix only partially processed (Critical)

**Severity:** Critical · **Effort:** L (requires redesign)
**Files:** `gated_delta_net_chunk.cu:21-24, 46-52, 213-220`

### Problem

The block dimensions are `(warp_size, (S_v + warp_size - 1) / warp_size, 1)`, giving `blockDim.y = 2` for `S_v=128`. Each `(row_y, lane)` pair owns `rows_per_lane` elements of one state "row." With 2 row_y values × 128 elements each = 256 state elements processed per block. The state matrix is `S_v × S_v = 16384` elements for `S_v=128` — **98.4% of the state is never touched**.

State load at line 49:
```cpp
s_shard[r] = (i < S_v) ? state_in[row_y * S_v + i] : 0.0f;
```
For `row_y ∈ {0, 1}` this reads only `state_in[0..127]` and `state_in[128..255]`, i.e., the first two rows of the 128×128 state matrix.

State write at line 216:
```cpp
state_out[row_y * S_v + i] = s_shard[r];
```
Same limitation — only 2 rows written.

The per-token kernel (`gated_delta_net.cu:42-44`) solves this correctly by using `gridDim.z = (S_v + num_warps - 1) / num_warps` to distribute columns across multiple blocks, with `col = blockIdx.z * blockDim.y + threadIdx.y`.

### Impact

Every model with `S_v > warp_size × blockDim.y` will have corrupted state after the first chunk. Since `blockDim.y = ceil(S_v / warp_size)` and each warp handles one row, the kernel only processes `ceil(S_v / warp_size)` rows out of `S_v`. For `S_v=128`: 2 of 128 rows.

### Fix

The chunked kernel needs either:
1. A `gridDim.z` dimension to partition state columns across blocks (matching the per-token kernel's approach), or
2. An internal loop over state columns within each block.

This is a fundamental architectural change — the current single-block-per-(head,seq) design cannot handle the full state for any `S_v > warp_size`.

## Finding 4 — Forward solve has within-column data race (High)

**Severity:** High · **Effort:** M
**File:** `gated_delta_net_chunk.cu:131-143`

```cpp
for (int col = 0; col < CS; col++) {
    if (lane < CS && lane >= col) {
        const int i = lane;
        float acc = (i == col) ? 1.0f : -L_smem[i * CS + col];
        for (int r = col; r < i; r++) {
            acc -= L_smem[i * CS + r] * A_smem[r * CS + col];

        A_smem[i * CS + col] = acc;

    __syncwarp();

```

Forward substitution is inherently sequential along rows: `A[i, col]` depends on `A[r, col]` for `r < i`. All lanes compute their row simultaneously within a column. Lane `i` reads `A_smem[r * CS + col]` which lane `r` writes at the **end** of its own computation — but on GCN, all lanes execute the write instruction at the same point in the instruction stream (after the inner loop completes for all lanes). Lane `i > r` reads `A[r, col]` during its inner loop iteration `r`, before lane `r` has executed the write.

On gfx906 (GCN wavefront), shared memory writes within a wavefront are visible to subsequent instructions in the same wavefront, but the loop structure here means the read of `A[r, col]` by lane `i` happens **before** the write by lane `r` in the instruction sequence.

### Fix

Process rows sequentially within each column:
```cpp
for (int col = 0; col < CS; col++) {
    for (int i = col; i < CS; i++) {
        if (lane == 0) {
            float acc = (i == col) ? 1.0f : -L_smem[i * CS + col];
            for (int r = col; r < i; r++) {
                acc -= L_smem[i * CS + r] * A_smem[r * CS + col];
           _smem[i * CS + col] = acc;

        __syncwarp();

```
This serializes the solve (only lane 0 does work) but is correct. For CS=64 this is 64×64/2 = 2048 sequential ops — acceptable since it's not the bottleneck. Alternatively, use a parallel triangular solve algorithm (e.g., block-recursive), but sequential is simplest and most reliable for CS≤64.

## Finding 5 — v_new_smem written by all lanes, only lane 0 correct (High)

**Severity:** High · **Effort:** S
**File:** `gated_delta_net_chunk.cu:150-178`

```cpp
for (int p = 0; p < CS; p++) {
    // ... k_cd_p computed via warp_reduce_sum (same on all lanes) ...
    float v_new_p = v_reg_local[p] - k_cd_p;
    v_new_smem[p] = v_new_p;

```

`v_reg_local[p]` is only initialized for `lane == 0` (line 77). For other lanes it is uninitialized (the `else` branch at line 84 only initializes it for lane 0). All lanes write `v_new_smem[p]`, but only lane 0's value is correct. After `__syncthreads()`, subsequent reads of `v_new_smem[p]` may see garbage from a non-zero lane.

### Fix

Either guard the write:
```cpp
if (lane == 0) v_new_smem[p] = v_new_p;
```
Or initialize `v_reg_local` for all lanes (broadcast from lane 0 via `__shfl`):
```cpp
if (lane == 0) v_reg_local[p] = v_t[row_y * warp_size];
v_reg_local[p] = __shfl(v_reg_local[p], 0); // broadcast from lane 0
```
