## 1. Correctness Review

### Finding 1 — CRITICAL: `lane == 0` guards leave other lanes with uninitialized registers

**Severity:** Critical (silent wrong-answer, non-deterministic, model-level corruption)
**File:** `ggml/src/ggml-cuda/gated_delta_net_chunk.cu:100-130`
**Effort:** XS

The load phase writes `v_reg[p]`, `beta_reg[p]`, and `g_reg[p]` (non-KDA) only on lane 0, but the processing phase (lines 145–178) reads them on **every lane** to compute `delta_col` and update `s_shard[r]`. Registers are per-thread; non-lane-0 threads read uninitialized memory. This corrupts the state recurrence and the attention output.

Evidence — load side:

### Finding 2 — Low: spurious `__syncthreads()`

**Severity:** Low (perf/clarity, not correctness)
**File:** `gated_delta_net_chunk.cu:136, 178`
**Effort:** XS

The kernel uses no shared memory; data flows through registers and warp shuffles. The two `__syncthreads()` calls synchronize four warps for nothing. They're not incorrect, but they cost ~100–500 ns each per chunk and signal to readers that shared state exists when it doesn't.

**Fix:** delete both, or replace with a comment explaining the design choice if kept defensively.

### Finding 3 — Informational: stride/index formula shared with per-token kernel

Both kernels compute `v + sequence*sv3 + t*sv2 + h_idx*sv1` and then read `[col]`. ## 3. Numerical Tolerance

The chunked kernel is documented to be "bit-for-bit equal to the per-token kernel when run over the same tokens" because both use the same FP32 recurrence order with no chunk-level reductions that would reorder additions. Empirically for this class of kernel:

| Comparison | atol | rtol | Notes |
|---|---|---|---|
| Chunked vs per-token (both CUDA, FP32) | 0 | 0 | Should be bitwise identical after Finding 1 is fixed |
| Chunked vs CPU reference | 1e-4 | 1e-4 | CPU side uses ggml ops that may reduce in a different order |
| Chunked vs CPU reference (KDA, long seq ≥256) | 5e-4 | 5e-4 | `expf(g)` accumulates more rounding divergence |
| Chunked vs CPU (S_v = 128, n_tokens ≥ 512) | 1e-3 | 1e-3 | Long recurrence, acceptable |

Use a two-tier comparison: first assert bitwise equality against the per-token CUDA kernel (this is the tightest, most meaningful invariant), then a looser check against CPU.

**Do not use cosine similarity** for this kernel — a single corrupted state slot can still yield cos_sim > 0.999 while being semantically wrong.
