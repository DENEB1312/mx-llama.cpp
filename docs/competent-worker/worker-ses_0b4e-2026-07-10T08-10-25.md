# Code Review: TP AllReduce NT-load + block-count changes

## 1. `load1_nt` for `nv_bfloat16` — out-of-bounds + misaligned read (HIGH)

This is the one real defect in the diff.

```cpp
static __device__ __forceinline__ float load1_nt(const nv_bfloat16 * p) {
    const uint32_t w = __builtin_nontemporal_load(reinterpret_cast<const uint32_t *>(p));
    return bf16_raw_to_f32((uint16_t)(w & 0xFFFF));
}
```

Two problems:

**(a) 2-byte out-of-bounds read at the tail.** The remainder loop (`k_broadcast_reduce` and `k_twoshot` stage 4) reads peer `r`'s last element at element offset `r*n_elements + (n_elements-1)` (in bf16 units). A 32-bit load reads the 4 bytes starting there, so the transaction reaches byte `2*((r+1)*n_elements) + 1`. For `r == nranks-1` this is **2 bytes past the end of the staging allocation** (`2*nranks*n_elements`). The value is masked out so *output is unaffected*, but it's a genuine OOB device-memory read — a robustness/UB bug, and on a platform with strict misalignment or guard-page semantics it could fault.

**(b) Misaligned 32-bit load.** The base byte offset is `2*(r*n_elements + idx)`. When `r*n_elements + idx` is odd (trivially reachable, e.g. `n_elements` odd and odd `idx` in the remainder), the `uint32_t*` access is only 2-byte aligned. A 32-bit NT load on a 2-byte-aligned address is technically UB; on AMD it usually works but is slower and less safe than a 16-bit load.

**Fix (removes both issues — the value is still correct, alignment is always 2-byte, no OOB):**

```cpp
static __device__ __forceinline__ float load1_nt(const nv_bfloat16 * p) {
#if defined(GGML_USE_HIP)
    const uint16_t w = __builtin_nontemporal_load(reinterpret_cast<const uint16_t *>(p));
    return bf16_raw_to_f32(w);
#else
    return wire_to_f32<nv_bfloat16>(*p);
#endif
}
```

The float `load1_nt` is fine (4-byte element, read ends exactly at buffer end).

## 4. Block-count increase (36→120 max, HIP 16→36 default) — beneficial, minor waste

- **`kDefaultBlocks` 16→36 on HIP is a real fix, not just a tweak.** MI50 (gfx906) has 32 CUs. With `__launch_bounds__(512,1)` a 512-thread block fills a CU's 8-wave capacity, so ≥32 blocks are needed to keep all CUs busy. A default of 16 left ~half the GPU idle. 36 ≥ 32, so this is correct and important.
- **`kMaxBlocks=120` is oversized.** With `launch_bounds(512,1)` max concurrent blocks ≈ 32 (one/CU), so 120 only over-subscribes the queue. Harmless but wastes Signal memory: `Signal` is now `120*16*4*2 + 120*4 ≈ 15.8 KB` × nranks (×8 = ~126 KB uncached). 64 would be plenty. Not a bug, just generous.
- **Occupancy/register pressure:** register budget per wave with `launch_bounds(512,1)` is 256 VGPR / 8 waves = 32 VGPR/wave; the new NT loads don't meaningfully change that. The block bump doesn't reduce occupancy — it only enlarges the grid, which is what you want here. No concern.
